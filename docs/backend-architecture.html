このページは docs/design.md の第10章を、実装時に参照しやすいようにフロー図と層の責務へ再構成した補助資料です。一次ソースは docs/design.md です。フロントエンド側は frontend-architecture.html を参照してください。

基本方針

apps/api（Hono / Cloudflare Workers）は route → service → dal の 3 層 に分けます。層を分ける目的はただ一つ、「HTTP の関心事」「業務の手順」「SQL の関心事」を混ぜないことです。

route は HTTP の入口

routes/ はパス・メソッド・zValidator による入力検証・ステータスコード・レスポンス整形を扱います。ビジネスロジックは書きません。

service はユースケース

services/ は「採点 → 記録 → SRS 更新」のような業務手順の一連です。Hono・D1・Drizzle に依存しない純 TypeScript なので、DB なしでテストできます。

dal は永続化アクセス

dal/ は Drizzle クエリの置き場です。upsert や db.batch のような SQL 的関心事をここに閉じ込め、service には漏らしません。

  • 依存方向は routes → services → dal →（D1）の一方向のみ。逆方向の import は禁止。packages/shared はどの層からも参照可。
  • service が「必要な関数の型（AnswerDeps など）」を宣言し、dal がそれを実装する。route が両者を接続する。DI コンテナは使わず、関数引数で依存を渡す。
  • 採点の権威は API（design.md §7.2）。クライアントは selectedIndex を送るだけで、正解も user_id も信用しない。
  • user_id は middleware（user-context.ts）が注入する。固定値の知識はこの 1 ファイルに閉じ、将来の認証導入時はここだけ差し替える。
  • 時刻（now）は service 内で取得せず、route から引数注入する。sm2 純粋関数と同じ設計で、テストで時刻を固定できる。
  • D1 は対話型トランザクションが使えないため、複数テーブルへの書き込みは db.batch() で原子的にまとめる。
  • hc の型推論を保つため、ルート定義は必ずメソッドチェーンで書き、index.ts で .route() のチェーンに合成して AppType をエクスポートする。
なぜ feature 型（縦割り）にしないのか 本アプリの中核 API（3 エンドポイント）はいずれも SRS という同じデータ（questions・answer_logs・srs_states）の周りを回っており、feature ごとにフォルダを切っても repository を相互 import することになります。境界が曖昧な縦割りは置き場所の迷いを生むため、層ごとの横割り＋依存方向ルールを採用しています。後続の集計系（アナリティクス・領域一覧）を含め独立したドメインが増えた時点で feature 型への移行を再検討します（移行はファイル移動のみで、3 層の責務はそのまま持ち越せます）。

API 契約（design.md §7.3）

本資料が詳細化するのは Walking Skeleton 中核の 3 エンドポイント（POST /answers・GET /review/queue・GET /dashboard/due-count）です。本文（教材・問題・解説）はファイル一次ソースのため、API は ID・SRS メタ・集計値のみ返します。

エンドポイント リクエスト → レスポンス 役割
POST /answers { questionId, selectedIndex, responseTimeMs? } → { isCorrect, correctIndex } 採点（D1 の questions と照合）→ answer_log 記録（任意で反応時間も）→ SRS 更新
GET /review/queue → { items: [{ questionId, dueAt }] } due 問題の収集（本文はフロントがバンドル済み content から join）
GET /dashboard/due-count → { dueCount } ダッシュボードの due 件数

リクエスト／レスポンスの Zod スキーマは packages/shared/src/schema/api.ts に置き、「API 入力の zValidator」「route 返り値の satisfies」「フロントの型（z.infer）」の 3 経路で共有します（§10.6）。

後続エンドポイント（Walking Skeleton 貫通後） §7.3 は上記 3 本に加えて画面駆動の集計 API を挙げます：GET /domains（領域別の習得率・トピック/レッスン数）、GET /analytics/summary・GET /analytics/weekly・GET /analytics/mistakes（アナリティクス画面の各集計）、GET /activity/recent（最近のアクティビティ）。これらは Walking Skeleton 貫通後に同じ route → service → dal パターンで追加します（§10.1）。集計クエリ主体（読み取りのみ）のため service 層は薄くなります。

判断基準

迷った時は「どの層の関心事か」で置き場所を決めます。HTTP の言葉（ステータスコード・ヘッダー）が出てきたら route、業務の言葉（採点・復習）が出てきたら service、SQL の言葉（upsert・batch・where）が出てきたら dal です。

迷う処理 置き場所 理由
リクエストボディの形の検証 routes/（zValidator） HTTP 契約そのものだから。スキーマ定義は shared に置く。
「正解かどうか」の判定 services/answer-service.ts 採点はユースケースの中心。HTTP や SQL とは無関係。
次回出題日（SRS）の計算 packages/shared/src/srs/sm2.ts 純粋関数として実装済み。service は呼ぶだけで、計算式を持たない。
srs_states の upsert の書き方 dal/answer-repository.ts onConflictDoUpdate・db.batch は SQL 的関心事だから。
エラーを 404 / 500 に写像する index.ts の app.onError HTTP ステータスは route 層の横断処理。service はドメインエラーを throw するだけ。
固定 user_id の値 middleware/user-context.ts 将来の認証差し替えポイントを 1 箇所に閉じるため（§10.4）。
リクエスト／レスポンスの型定義 packages/shared/src/schema/api.ts フロントと API の契約なので、単一ソースで共有するため。
Date.now() の呼び出し routes/（service へ引数注入） service を純粋に保ち、テストで時刻を固定できるようにするため。
採用する考え方

service は「deps という関数の束」を受け取る。DB を知らないので、テストは素朴な fake オブジェクトで書け、モックライブラリも Workers ランタイムも要らない。

避ける考え方

route ハンドラに採点・SRS 更新・Drizzle クエリを直接書く。テストに workerd が必須になり、書き込みの原子性（batch）も漏れやすい。

コード例

以下は実装イメージです。実コードでは import path や型名を既存ファイルに合わせます。

service: DB を知らないユースケース

deps 宣言 + 純 TS 
// apps/api/src/services/answer-service.ts
// Hono・D1・Drizzle に依存しない純 TS。deps は「必要な関数の束」として宣言する
import { initialSrs, reviewSrs, type SrsInput, type SrsResult } from '@tsl/shared';
import { QuestionNotFoundError } from './errors';

export type AnswerDeps = {
  findAnswerIndex(questionId: string): Promise<number | null>;
  findSrsState(userId: string, questionId: string): Promise<SrsInput | null>;
  recordAnswer(params: {
    userId: string;
    questionId: string;
    isCorrect: boolean;
    answeredAt: number;
    nextSrs: SrsResult;
  }): Promise<void>;
};

export type SubmitAnswerInput = {
  userId: string;
  questionId: string;
  selectedIndex: number;
  now: number; // 時刻は service 内で取得せず引数注入（テストで固定できる）
};

export async function submitAnswer(deps: AnswerDeps, input: SubmitAnswerInput) {
  const answerIndex = await deps.findAnswerIndex(input.questionId);
  if (answerIndex === null) throw new QuestionNotFoundError(input.questionId);

  const isCorrect = answerIndex === input.selectedIndex; // 採点権威は API（§7.2）
  const current =
    (await deps.findSrsState(input.userId, input.questionId)) ?? initialSrs();
  const nextSrs = reviewSrs(current, isCorrect, input.now); // 純粋関数（sm2）

  await deps.recordAnswer({
    userId: input.userId,
    questionId: input.questionId,
    isCorrect,
    answeredAt: input.now,
    nextSrs,
  });

  return { isCorrect, correctIndex: answerIndex };
}

route: 検証と接続だけを担当する

zValidator + deps 生成 + now 注入 
// apps/api/src/routes/answers.ts
import { Hono } from 'hono';
import { zValidator } from '@hono/zod-validator';
import { drizzle } from 'drizzle-orm/d1';
import { answerRequestSchema, type AnswerResponse } from '@tsl/shared';
import type { AppEnv } from '../env';
import { createAnswerDeps } from '../dal/answer-repository';
import { submitAnswer } from '../services/answer-service';

// hc の型推論のため、new Hono() からのメソッドチェーンで定義する
export const answersRoute = new Hono<AppEnv>().post(
  '/',
  zValidator('json', answerRequestSchema),
  async (c) => {
    const { questionId, selectedIndex } = c.req.valid('json');
    const result = await submitAnswer(createAnswerDeps(drizzle(c.env.DB)), {
      userId: c.get('userId'), // middleware が注入済み（§10.4）
      questionId,
      selectedIndex,
      now: Date.now(),
    });
    return c.json(result satisfies AnswerResponse); // 契約は shared の型で固定
  },
);

dal: Drizzle と db.batch を閉じ込める

AnswerDeps の実装 
// apps/api/src/dal/answer-repository.ts
import { and, eq } from 'drizzle-orm';
import type { DrizzleD1Database } from 'drizzle-orm/d1';
import { db as schema } from '@tsl/shared';
import type { AnswerDeps } from '../services/answer-service';

export function createAnswerDeps(db: DrizzleD1Database): AnswerDeps {
  return {
    async findAnswerIndex(questionId) {
      const rows = await db
        .select({ answerIndex: schema.questions.answerIndex })
        .from(schema.questions)
        .where(eq(schema.questions.questionId, questionId));
      return rows[0]?.answerIndex ?? null;
    },

    async findSrsState(userId, questionId) {
      const rows = await db
        .select()
        .from(schema.srsStates)
        .where(
          and(
            eq(schema.srsStates.userId, userId),
            eq(schema.srsStates.questionId, questionId),
          ),
        );
      return rows[0] ?? null;
    },

    async recordAnswer({ userId, questionId, isCorrect, answeredAt, nextSrs }) {
      const srsRow = {
        ease: nextSrs.ease,
        intervalDays: nextSrs.intervalDays,
        dueAt: new Date(nextSrs.dueAt),
        reps: nextSrs.reps,
        lapses: nextSrs.lapses,
      };
      // D1 は対話型トランザクション不可 → db.batch で原子的に書く（§10.7）
      await db.batch([
        db.insert(schema.answerLogs).values({
          id: crypto.randomUUID(),
          userId,
          questionId,
          isCorrect,
          answeredAt: new Date(answeredAt),
        }),
        db
          .insert(schema.srsStates)
          .values({ userId, questionId, ...srsRow })
          .onConflictDoUpdate({
            target: [schema.srsStates.userId, schema.srsStates.questionId],
            set: srsRow,
          }),
      ]);
    },
  };
}

index.ts: 合成・onError・AppType

メソッドチェーン必須 
// apps/api/src/index.ts
import { Hono } from 'hono';
import { cors } from 'hono/cors';
import type { AppEnv } from './env';
import { userContext } from './middleware/user-context';
import { QuestionNotFoundError } from './services/errors';
import { answersRoute } from './routes/answers';
import { reviewRoute } from './routes/review';
import { dashboardRoute } from './routes/dashboard';

const app = new Hono<AppEnv>();

// CORS はブラウザ経路（§3.1）用。Service Binding 経由には関与しない
app.use('*', async (c, next) => cors({ origin: c.env.WEB_ORIGIN })(c, next));
app.use('*', userContext);

// service のドメインエラーを HTTP へ写像する（レスポンス形を統一）
app.onError((err, c) => {
  if (err instanceof QuestionNotFoundError) {
    return c.json({ error: { code: 'QUESTION_NOT_FOUND', message: err.message } }, 404);
  }
  console.error(err); // Workers Logs で観測
  return c.json({ error: { code: 'INTERNAL', message: 'Internal Server Error' } }, 500);
});

// hc の型推論を保つため、必ずメソッドチェーンで合成する
const routes = app
  .get('/health', (c) => c.json({ status: 'ok' as const }))
  .route('/answers', answersRoute)
  .route('/review', reviewRoute)
  .route('/dashboard', dashboardRoute);

export type AppType = typeof routes;
export default app;

避ける例: route にすべてを書く

責務が混ざる例 
// 避ける: route ハンドラが検証以外のすべて（採点・SRS・SQL）を知っている
export const answersRoute = new Hono<AppEnv>().post(
  '/',
  zValidator('json', answerRequestSchema),
  async (c) => {
    const { questionId, selectedIndex } = c.req.valid('json');
    const db = drizzle(c.env.DB);

    const question = (
      await db.select().from(schema.questions)
        .where(eq(schema.questions.questionId, questionId))
    )[0];
    if (!question) return c.json({ error: 'not found' }, 404);

    const isCorrect = question.answerIndex === selectedIndex;
    const state = (await db.select().from(schema.srsStates) /* ... */)[0] ?? initialSrs();
    const next = reviewSrs(state, isCorrect, Date.now());

    await db.insert(schema.answerLogs).values({ /* ... */ });  // batch されていない！
    await db.insert(schema.srsStates).values({ /* ... */ });   // 途中失敗で不整合が残る

    return c.json({ isCorrect, correctIndex: question.answerIndex });
  },
);
この例には 3 つの問題があります。①採点・SRS のテストに実 DB（workerd）が必須になる、②2 つの書き込みが db.batch にまとまっておらず途中失敗で不整合が残る、③採点ロジックを他の場所（将来の記述式問題など）から再利用できない。services と dal に分けるとこの 3 つが同時に解消されます。

テスト戦略（design.md §10.9）

対象 方法 ランタイム
SRS（sm2） 純粋関数の単体テスト（実装済み） Node
service deps をインメモリ fake に差し替えた単体テスト。採点の正誤・SRS 遷移の呼び出し・エラー系（問題未存在）を重点 Node
route + dal @cloudflare/vitest-pool-workers（ローカル D1 に対する実クエリ）で happy path を最低 1 本（POST /answers の貫通） workerd

service の deps を「フラットな関数の束」にしているのはこのためです。fake は素朴なオブジェクトリテラルで書け、モックライブラリを必要としません。テストのピラミッドは「純粋関数（多）→ service（中）→ 実 D1 貫通（少）」の順に薄くなります。

フロントエンドとの境界

API 契約は AppType（hc の型安全 RPC）と packages/shared/src/schema/api.ts の 2 点で共有され、OpenAPI は使いません。web からの呼び出し経路は frontend-architecture.html の「Worker 構成と接続」を参照してください（Server loader は Service Binding、Client hook は公開 URL。どちらも同じ hc<AppType> です）。

実装時の注意（design.md §10.10） 本資料のコードが前提とする差分：questions テーブルの追加、srs_states の複合主キー (user_id, question_id)、answer_logs への任意列 response_time_ms 追加、lesson_views テーブルの追加（アナリティクス用）、schema/api.ts の新設、wrangler の Worker 名を tech-study-lab-api へ変更（web 側 Service Binding が参照する名前）、vars への WEB_ORIGIN 追加。集計系エンドポイント（/domains・/analytics/*・/activity/recent）の route/service/dal は Walking Skeleton 貫通後に同パターンで追加。
