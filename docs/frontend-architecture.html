基本方針

app は URL の入口

src/app は App Router の規約を置く場所です。URL、layout、metadata、error、route params を扱い、feature の loader に引き渡します。

features は画面価値の単位

features は lesson、quiz、review のような機能単位です。表示部品、ViewModel、mapper、初回取得、再取得を近くに置きます。

lib は横断インフラ

lib は API client、response helper、content index など feature 非依存の関数だけを置きます。画面都合の整形は置きません。

  • apps/web/src/app は routing と page 組み立てに限定し、ロジックは apps/web/src/features に寄せる。
  • 教材・問題は content/ を一次ソースとし、RSC / Server loader からビルド時バンドル済みデータを読む。
  • 解答ログ・SRS・due queue・各種集計（ダッシュボード統計・/domains の習得率・/analytics の集計）などの動的データだけを Hono API に集約し、フロントからは hc の型安全RPCで呼ぶ。集計系ページも RSC で、初回表示に必要な集計値を Server loader から読む（教材本文ページのみ API 不要）。
  • 演習・復習は 1 ルート内でクライアント状態遷移（intro → exercise → result）を持つ（別 URL に切らない）。初回データ （問題・解説、/review は due queue）は Server loader で ViewModel 化して props 渡しし、Client Component が画面フェーズ・解答結果・送信エラーだけを state に持つ。
  • web と API は別 Worker（design.md §3.1）。Server loader からは Service Binding （env.API）で内部呼び出し、Client hook からは API Worker の公開 URL で呼ぶ。どちらの経路でも hc<AppType> の型は同一。
  • 初回表示は Server loader で ViewModel 化して props 渡し、Client Component はその VM をそのまま描画に使う（state に複製しない）。Client hook が持つのは「解答結果」「送信エラー」だけ。
  • due queue の再取得は Client 側で content join をやり直すのではなく、router.refresh() で Server Component を再実行し、Server loader が新しい VM を作り直す。
  • 正誤判定は API（サーバー）が権威。VM は選択肢と解説のみを持ち、正解 （answerIndex）を含めない。
  • 動的データを読むページ（/（due-count・統計）・/domains（習得率）・/analytics（各集計）・/review（queue））は無キャッシュ fetch により route segment config の既定（dynamic = 'auto'）のまま 自動的に動的レンダリングになる。'use cache'・cacheTag・revalidateTag は現時点では 採用しない。
ディレクトリ名だけで実行場所は決まりません。apps/web/src/features/*/server は apps/web/src/app/**/page.tsx など Server Component から import されることでサーバー側で実行されます。誤 import 防止として import 'server-only' を置きます。

判断基準

迷った時は「どこから呼ばれるか」ではなく「何の責務か」で置き場所を決めます。

迷う処理 置き場所 理由
route params から lessonId を取り出す app/**/page.tsx App Router の URL 境界そのものだから。
lesson content を読み、LessonViewModel に変換する features/lesson/server lesson feature の初回データ形成だから。page に置くと page が厚くなる。
review queue DTO と content question を join する features/review/server + features/review/mapper.ts review 画面の表示要件に合わせた ViewModel 化だから。
client.review.queue.$get() を呼ぶ features/review/api Hono RPC の endpoint wrapper として feature 単位に閉じるため。
res.ok チェックと res.json() lib/api-response.ts 全 endpoint で同じ横断処理だから。
ボタン、カード、入力欄などの汎用見た目 components/ui feature と無関係に再利用できる UI だから。
採用する考え方

app は薄い controller、features/*/server は Server 専用 loader、mapper は純粋変換として分ける。

避ける考え方

「page からしか呼ばないから page に全部書く」と判断する。これは URL 境界と機能ロジックを混ぜる原因になる。

ディレクトリの役割

パス 役割 置かないもの
apps/web/src/app App Router の page / layout / error。Server loader を呼び、ViewModel を component に渡す薄い入口。 DTO変換、APIレスポンス処理、複雑なUI state。
apps/web/src/features/*/components feature 固有の表示コンポーネント。必要に応じて Client Component とする。 DBアクセス、Hono route 直書き、content の直接走査。
apps/web/src/features/*/hooks Client 専用。mutation、再取得、loading/error state、UI state を扱う。 初回取得の必須データ形成、server-only import。
apps/web/src/features/*/server Server loader。初回取得、content/API DTO の取得、mapper 呼び出し、ViewModel 生成。 Client Component からの import、ブラウザ専用API。
apps/web/src/features/*/api hc の endpoint wrapper。path・method・引数・エラーメッセージだけを持つ。 ViewModel 化、UI state、cookies/headers など環境専用処理。
apps/web/src/features/*/mapper.ts Content data / DTO を ViewModel へ変換する純粋関数。 fetch、DBアクセス、副作用。
apps/web/src/features/*/view-model.ts web 表示都合の型。page/component がそのまま使える形を定義する。 API契約の一次定義。DTO は packages/shared 由来。
apps/web/src/features/shared contentQuestionToQuizQuestion のような小さい純粋変換。 共通 ViewModel、loader、hook、feature 全体の抽象化。
apps/web/src/components/ui Button / Card / Badge / ProgressBar / TermWin / Keycap などの汎用UI。Tailwind ＋ Dev-Native Neo Flat × Terminal デザイントークン（--bg / --surface / --well / --border 等、design.md §8.7）を土台に集約する。 feature 固有ロジック、データ取得。
apps/web/src/lib/api.ts createServerApiClient（本番は Service Binding の fetch、dev は URL フォールバック）/ createBrowserApiClient（公開 URL）の Hono client 生成。 endpoint ごとの business wrapper。
apps/web/src/lib/api-response.ts requestJson。res.ok チェックと res.json() を共通化する。 Hono path の文字列汎用 fetch 化。
apps/web/src/lib/content.ts ビルド時バンドル済み content の参照口。lesson と question index を提供する。 feature 固有 ViewModel への変換。

コード例

以下は実装イメージです。実コードでは import path や型名を既存ファイルに合わせます。

教材ページ: page は loader を呼ぶだけ

app → features/server → mapper 
// apps/web/src/app/learn/[domain]/[topic]/[lesson]/page.tsx
import { LessonDisplay } from '@/features/lesson/components/lesson-display';
import { loadLesson } from '@/features/lesson/server/load-lesson';

type Props = {
  params: {
    domain: string;
    topic: string;
    lesson: string;
  };
};

export default function LessonPage({ params }: Props) {
  const viewModel = loadLesson(params.lesson);

  return <LessonDisplay viewModel={viewModel} />;
}
// apps/web/src/features/lesson/server/load-lesson.ts
import 'server-only';

import { getLessonContent } from '@/lib/content';
import { lessonContentToViewModel } from '../mapper';
import type { LessonViewModel } from '../view-model';

export function loadLesson(lessonId: string): LessonViewModel {
  const content = getLessonContent(lessonId);
  return lessonContentToViewModel(content);
}
// apps/web/src/features/lesson/mapper.ts
import type { LessonContent } from '@tsl/shared/schema/content';
import type { LessonViewModel } from './view-model';

export function lessonContentToViewModel(content: LessonContent): LessonViewModel {
  return {
    id: content.id,
    title: content.title,
    markdownBody: content.markdownBody,
    questions: content.questions.map((question) => ({
      id: question.id,
      prompt: question.prompt,
      choices: question.choices,
    })),
  };
}

復習ページ: 初回は Server、操作後は Client

server loader + client hook 
// apps/web/src/features/review/api/review-api.ts
import type { ApiClient } from '@/lib/api';
import { requestJson } from '@/lib/api-response';
import type { ReviewQueueDTO } from '@tsl/shared/schema/review';

export function fetchReviewQueue(client: ApiClient): Promise<ReviewQueueDTO> {
  return requestJson(
    () => client.review.queue.$get(),
    'Failed to fetch review queue',
  );
}
// apps/web/src/features/review/server/load-review-queue.ts
import 'server-only';

import { createServerApiClient } from '@/lib/api';
import { getBundledQuestions } from '@/lib/content';
import { fetchReviewQueue } from '../api/review-api';
import { reviewQueueDTOToViewModel } from '../mapper';
import type { ReviewViewModel } from '../view-model';

export async function loadReviewQueue(): Promise<ReviewViewModel> {
  const dto = await fetchReviewQueue(createServerApiClient());
  return reviewQueueDTOToViewModel(dto, getBundledQuestions()); // join はサーバー側のみ
}
// apps/web/src/app/review/page.tsx
// 無キャッシュ fetch により dynamic = 'auto' のまま動的レンダリングになる
import { ReviewRunner } from '@/features/review/components/review-runner';
import { loadReviewQueue } from '@/features/review/server/load-review-queue';

export default async function ReviewPage() {
  const viewModel = await loadReviewQueue(); // API queue + content join 済み VM

  return <ReviewRunner viewModel={viewModel} />;
}
// apps/web/src/features/review/hooks/use-answer-submit.ts
'use client';

import { useCallback, useState } from 'react';
import { createBrowserApiClient } from '@/lib/api';
import { postAnswer } from '../api/review-api';
import type { AnswerResult } from '@tsl/shared/schema/api';

// VM は保持しない。解答結果とエラーだけを扱う mutation 専用 hook
export function useAnswerSubmit() {
  const [results, setResults] = useState<Record<string, AnswerResult>>({});
  const [error, setError] = useState<Error | null>(null);

  const submitAnswer = useCallback(async (questionId: string, selectedIndex: number) => {
    try {
      const result = await postAnswer(createBrowserApiClient(), { questionId, selectedIndex });
      setResults((prev) => ({ ...prev, [questionId]: result }));
    } catch (cause) {
      setError(cause instanceof Error ? cause : new Error('Failed to submit answer'));
    }
  }, []);

  return { results, error, submitAnswer };
}
// apps/web/src/features/review/components/review-runner.tsx
'use client';

import { useRouter } from 'next/navigation';
import { useAnswerSubmit } from '../hooks/use-answer-submit';
import type { ReviewViewModel } from '../view-model';

type Props = {
  viewModel: ReviewViewModel;
};

export function ReviewRunner({ viewModel }: Props) {
  const router = useRouter();
  const { results, error, submitAnswer } = useAnswerSubmit();

  const handleAllAnswered = () => {
    router.refresh(); // Server loader を再実行し、次の due queue を props で受け直す
  };

  if (error) {
    return <ErrorDisplay error={error} retry={() => router.refresh()} />;
  }

  return (
    <QuizRenderer
      viewModel={viewModel}
      results={results}
      onAnswer={submitAnswer}
      onComplete={handleAllAnswered}
    />
  );
}

避ける例: page に ViewModel 化を直接書く

責務が混ざる例 
// apps/web/src/app/review/page.tsx
// 避ける: page が API DTO、content join、sort、ViewModel 形状を知っている。
export default async function ReviewPage() {
  const client = createServerApiClient();
  const res = await client.review.queue.$get();
  const dto = await res.json();
  const questions = getBundledQuestions();

  const viewModel = {
    items: dto.items.map((item) => ({
      ...questions[item.questionId],
      dueAt: item.dueAt,
      stability: item.stability,
    })),
  };

  return <ReviewRunner viewModel={viewModel} />;
}
この例では page が Hono RPC、レスポンス処理、content join、ViewModel 形状をすべて知っています。変更影響が page に漏れるため、features/review/server と features/review/mapper.ts へ分けます。

API層との境界

apps/api/src/
├── routes/       # Hono route。HTTP契約と zValidator
├── services/     # ユースケース。SRS更新などの処理
└── dal/          # Drizzle / D1 への永続化アクセス

apps/web 側では _DAL / dal を置きません。DB に直接触る責務は apps/api 側に閉じ、フロントは Hono RPC wrapper 経由で DTO を受け取ります。

Worker 構成と接続（design.md §3.1・§8.4）

apps/web（OpenNext）と apps/api（Hono）は 別 Worker としてデプロイします。Next.js の Route Handler 上に Hono を載せる同居構成は採用しません（wrangler 設定の所有権・デプロイ独立性・§8.3 キャッシュ方針の前提を維持するため）。呼び出し元によって経路が変わりますが、 hc<AppType> の型安全RPCは両経路で同一です。

Server loader は Service Binding による内部呼び出し（公衆インターネット非経由・CORS 不要）、Client hook は公開 URL（当面は cors() で許可、将来はルート割当で同一オリジン化）です。

// apps/web/src/lib/api.ts
import { getCloudflareContext } from '@opennextjs/cloudflare';
import { createClient } from '@tsl/api/client';

// Server loader 用: 本番は Service Binding、dev は URL にフォールバック
export function createServerApiClient() {
  const { env } = getCloudflareContext();
  if (env.API) {
    // ホスト名はダミー（Service Binding はパスのみ使用）
    return createClient('https://api.internal', {
      fetch: env.API.fetch.bind(env.API),
    });
  }
  return createClient(env.API_BASE_URL); // next dev + wrangler dev 並走時
}

// Client hook 用: env から解決した公開 baseURL
export function createBrowserApiClient() {
  return createClient(process.env.NEXT_PUBLIC_API_BASE_URL);
}
web 側の wrangler 設定に "services": [{ "binding": "API", "service": "<API Worker 名>" }] を宣言します。apps/api/src/client.ts のファクトリは hc の第2引数（fetch オプション等）を受け渡せる形に拡張します。
