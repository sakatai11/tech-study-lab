実装前に確認する API 契約

画面とAPIの境界を、Jr.には「何を送って何が返るか」が分かる形で、SeniorにはHTTP・Zod・実装状態をすぐ追える形で整理しています。

この文書の位置づけ

画面構成とユースケースの一次ソースは 設計文書 §7.3 です。このページはHTTP入出力と実装状況を参照しやすくするための契約カタログです。

最初に読むなら、この3点

1. 状態を見る

scheduled は実装時に守る契約、planned はDTOが未確定です。

2. 成功だけでなくエラーを見る

入力不備、対象なし、内部エラーを先に把握すると、画面側の分岐を設計しやすくなります。

3. Zodを正とする

実装済みの契約は packages/shared/src/schema/api.ts をAPIとフロントで共有します。

共通規約

パスと形式

現時点ではAPIバージョンのプレフィックスはありません。リクエストとレスポンスはJSONで、本文のない GET にJSON bodyは送りません。

ユーザーコンテキスト

API middleware が固定の user_id を権威的に注入します。クライアントは user_id / userId を送らず、レスポンスにも含めません。

エラー形式

routeのエラーは原則 { error: { code, message } }。ただし、入力検証の400は現時点で zValidator の既定形式です。

実装状態

scheduled は予定契約で、routeの実装済みを意味しません。planned はレスポンスDTOとZodスキーマが未確定です。

Walking Skeleton の中核API

scheduled · Zod実装済みroute実装は #20 で予定しています。

POST

/answers

scheduled

1問の解答をサーバーで採点・記録し、SRS状態を更新します。演習・復習・誤答のみ再挑戦で契約は共通です。

リクエストJSON。answerRequestSchema: questionId は1文字以上、selectedIndex は0〜5の整数、responseTimeMs は任意の0以上の整数。
成功200 OK。answerResponseSchema: isCorrect: boolean、correctIndex: 0〜5の整数。
エラー400 入力不備。404 QUESTION_NOT_FOUND（同期済み questions に質問IDがない）。500 INTERNAL。
Zod実装済み: answerRequestSchema / answerResponseSchema

なぜAPIで採点するか: クライアントに正解を持たせず、APIが questions の正解と照合するためです。

GET

/review/queue

scheduled

復習対象の問題IDとSRSメタデータを返します。本文・選択肢・解説はフロントエンドがcontentから解決します。

リクエストbody・query parameterなし。
成功200 OK。reviewQueueResponseSchema: { items: Array<{ questionId, dueAt }> }。dueAt はUnixエポックミリ秒の整数で、昇順・最大20件です。
エラー500 INTERNAL。
Zod実装済み: reviewQueueResponseSchema
GET

/dashboard/due-count

scheduled

ダッシュボードに表示する、今すぐ復習できる問題数を返します。

リクエストbody・query parameterなし。
成功200 OK。dueCountResponseSchema: { dueCount: 0以上の整数 }。
エラー500 INTERNAL。
Zod実装済み: dueCountResponseSchema

Walking Skeleton 後に追加するAPI

planned · DTO未確定目的・成功ステータス・予定スキーマを先に共有し、フィールドは各実装Issueで共有Zodスキーマとして確定します。

後続Issueで必ず更新するもの

各APIの実装に着手するときは、返却値を共有Zodスキーマで確定し、このページの成功・エラー応答と実装状態も同じPRで更新します。APIのIF（URL、HTTPステータス、JSON形式、エラー形式）を実装だけで先に変えないでください。
