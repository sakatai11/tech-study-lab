# retro-refine — セッションの摩擦をハーネス改善に変換する

実行前に `.ai/runtime-compatibility.md` を全文読む。共通定義は `.ai/` だけを編集し、Claude/Codex のリンクも確認する。

タスク完了後の「振り返り → 恒久対策」を担当するスキル。**その場しのぎの修正ではなく、次回以降に効く形（スキル・ルール・設定）へ落とし込む**ことが目的。

引数 = 振り返り対象の指定（省略時は**直前のタスク／セッション**）。

## 起動の原則（opt-in）

- **毎タスクでは起動しない**。実際に摩擦（繰り返しエラー・明確な回り道・手戻り・指示の曖昧さ）があったタスクの後にだけ使う。
- 摩擦が無かった／些細だったセッションでは「反映すべき恒久対策なし」と結論して早期終了してよい。

## ステップ1: 摩擦の洗い出し（discovery）

直前のタスク／セッションを振り返り、摩擦の候補を列挙する。手がかり:

- **繰り返したエラー・リトライ**: 同じ失敗を2回以上繰り返した箇所（例: ツール呼び出し崩れ、コマンド失敗、権限確認の連発）。
- **回り道・手戻り**: 誤った前提で進めてやり直した、探索に無駄が多かった、方針を途中で覆した箇所。
- **指示の曖昧さ**: ユーザーに聞き直した／推測で進めて外した箇所。ルール化で減らせるか。
- **環境・サンドボックス制約**: ツールの制約に起因する失敗（例: codex サンドボックスの `listen EPERM`）。
- **規約からの逸脱**: AGENTS.md / design.md の規約を見落として指摘された箇所。

各候補について「何が起きたか / 根本原因 / 現状のハーネスに書いてあったか」を1-2行でメモする。

## ステップ2: 実害フィルタ（重要）

洗い出した候補を**実害の有無で選別する**。すべてを取り込まない ― 過剰なルール化はハーネスを肥大化させ、かえって守られなくなる。

各候補を下表で振り分ける:

| 判定 | 基準 | 扱い |
|---|---|---|
| **採用** | 再発したら実際に時間/品質を損なう。かつ再発しやすい（構造的原因がある） | ステップ3で恒久対策に落とす |
| **見送り** | 一度きり・偶発的、または既にハーネスに記載済みで運用で防げる | 理由を添えて見送る（記録のみ） |
| **過剰** | ルール化すると硬直化・肥大化する / エッジケースすぎる | 取り込まない |

**「一度起きた」だけでは採用しない**。「構造的に再発する」「実害がある」の両方を満たすものだけ採用する。判断に迷う場合はユーザーに諮る。

## ステップ3: 対策先のルーティング

採用した対策を、**適切なハーネス成果物**に割り当てる。対策の性質で振り分ける:

| 対策の性質 | ルーティング先 | 例 |
|---|---|---|
| 特定ワークフローの手順改善 | 該当スキル `.ai/skills/<name>/SKILL.md` | 「エージェント起動時は長文ブリーフをファイルに逃がす」を issue-dev-orchestrate に追記 |
| 新しい反復ワークフローの発生 | 新規スキル `.ai/skills/<new>/SKILL.md` と両ランタイムのリンク | 繰り返す定型作業をスキル化 |
| エージェントの挙動・責務の調整 | 該当エージェント `.ai/agents/<name>.md` | reviewer の観点追加 |
| プロジェクト全体の規約・原則 | `AGENTS.md`（または `CLAUDE.md`） | ブランチ命名・PR 規約の明確化 |
| 仕様・設計判断 | `docs/design.md` | 確定した設計値の記録（仕様駆動） |
| コマンド権限・自動化 | `.claude/settings.json` / hooks | 頻出コマンドの allow 追加、hook での自動チェック |
| スキル発見・入口 | `.claude/skills/<name>` / `.agents/skills/<name>` | `.ai/skills/<name>` を指すシンボリックリンクの追加・検証 |
| Codex のプロジェクト規約 | `AGENTS.md` / Codex の sandbox・approval 境界 | Claude 専用設定を Codex にも効くと誤認しないよう実行条件を明記 |

- 1つの対策が複数の成果物にまたがる場合は分割する。
- 共通本文は `.ai/` だけに置き、`.claude/` と `.agents/` に複製しない。
- **既存ファイルへの追記を優先**し、新規作成は本当に独立したワークフローのときだけにする。

## ステップ4: 適用（ユーザー承認後）

1. ルーティング結果を**採用/見送り/過剰の一覧＋各採用対策の適用先・差分案**としてユーザーに提示する。
2. **ハーネスファイルは無断で書き換えない**（開発フローを規定するため）。承認された対策のみ、重要度の高い順に適用する。
3. 適用は既存のスタイル・命名・粒度に合わせる（周辺の記述に馴染ませる）。
4. スキル/エージェント/hooks の新規作成・変更を伴う場合、ブランチを切って対応し、PR 作成まで行うかをユーザーに確認する（ハーネス変更もレビュー対象）。

## ステップ5: 検証 — skill-audit の連結（条件付き・推奨）

- **スキル / エージェント / hooks を追加・変更した場合のみ**、最終検証として利用可能なスキル機構で `skill-audit` を起動する。未発見なら `.ai/skills/skill-audit/SKILL.md` を全文読んで同じ手順を実施する。
- `AGENTS.md` / `CLAUDE.md` / `docs/design.md` の散文だけを更新した場合は skill-audit の監査対象外なので**呼ばない**。
- これは**推奨ステップ（任意）**。ユーザーが省略を選べる。skill-audit は独立スキルのまま連結して使う（統合はしない）。

## 完了報告

- 洗い出した摩擦の件数と、採用/見送り/過剰の内訳。
- 適用したハーネス変更の一覧（ファイル・要点）。
- skill-audit を実行した場合はその結果。
- ブランチ・PR を作成した場合はその情報。

## やらないこと

- 摩擦が無いのに無理やり対策を作らない（空振りの振り返りは「対策なし」で終える）。
- その場限りのバグ修正やコード実装そのものは対象外（それは通常の実装フロー／issue-dev-orchestrate の担当）。retro-refine は「次回に効く恒久対策」だけを扱う。
